# Why I'm building this

I've spent the last decade building geospatial software: mapping tools,
spatial databases, address-matching pipelines, mostly for people who needed
a map to answer a question, not to make one. Along the way I kept running
into the same problem from a different angle: small health departments and
rural jurisdictions doing real epidemiology without the tooling that comes
standard at a state health department or a university. A GIS license is
expensive. SaTScan is free but has its own learning curve. A cloud
geocoding API means shipping patient addresses to a third party, which is a
nonstarter for a lot of health data.

I'm an MPH candidate in epidemiology now, finishing up a practicum with a
small rural health department, and MESA came out of watching that gap up
close: a jurisdiction with real disease clusters worth finding and no easy
way to find them without either paying for tools built for a much bigger
budget, or building something themselves.

So that's what this is. Offline geocoding and spatial cluster detection,
built to run on one laptop with one dataset, validated the way I'd want a
tool validated if I were the one trusting its output for a real
investigation. Not because I think everyone else is doing it wrong. Mostly
because I've been the analyst staring at a shapefile at 11pm wishing the
tool in front of me was simpler, and more honest about what it does and
does not know.

This notes section is where the "why" and the "how it went" will live,
since the methods pages are only supposed to describe what the code
actually does today. Some entries will be validation results. Some will be
"I tried this, it did not work, here is why." No schedule. I write one
when there is something worth saying.

If you are doing epidemiology somewhere without a GIS department behind
you, this is for you first. Everyone else is welcome to read along.
